> **質問:** RNNとこの教科書（生成AIメモリ技術入門）の内容の関連性はないのか？

### RNNと生成AIメモリ技術の関連性

#### 結論

教科書（生成AIメモリ技術入門）がRNNに言及しないのは、**TransformerがRNNの持っていた"メモリ"を捨てた**からである。教科書の内容はすべて「捨てたメモリをどう取り戻すか」という話に帰結する。

---

#### 1. RNNは「メモリを内蔵していた」アーキテクチャ

RNNの隠れ状態の更新式：

$$h_t = f(W_h \cdot h_{t-1} + W_x \cdot x_t)$$

- 前の時刻の状態 $h_{t-1}$ を次の計算に渡す → これが「記憶」
- LSTM/GRUはゲート機構で「何を覚え、何を忘れるか」を学習
- **RNNはアーキテクチャ自体がステートフル（状態を持つ）** だった

---

#### 2. Transformerは「メモリを捨てて並列性を得た」

教科書 第1章（1.1節）より：

> Transformerアーキテクチャは、本質的に**ステートレス**なシステムとして設計されている

RNNの最大の弱点は**逐次処理**であり、Transformerはメモリを犠牲にしてスケーラビリティを手に入れた。

|                  | RNN                                       | Transformer                          |
| :--------------- | :---------------------------------------- | :----------------------------------- |
| **処理方式**     | $t=1, 2, 3...$ と順番に計算（並列化不可） | 全トークンを同時に処理（並列化可能） |
| **記憶の仕組み** | 隠れ状態 $h_t$ に暗黙的に保持             | なし（コンテキストウィンドウ内のみ） |
| **長期依存性**   | 勾配消失で実質数十〜数百トークン          | Attention機構で窓内は等距離アクセス  |
| **学習速度**     | 遅い                                      | GPU並列化で圧倒的に速い              |

---

#### 3. 教科書の各章は「RNNが持っていたものを別の方法で再構築する試み」

| 教科書の章                                                 | RNNとの対応関係                                                               |
| :--------------------------------------------------------- | :---------------------------------------------------------------------------- |
| **第2章** コンテキストウィンドウの拡張                     | RNNの「理論上無限の系列長」をTransformerで再現しようとする試み                |
| **第3章** 対話履歴の管理（要約・スライディングウィンドウ） | RNNの隠れ状態 $h_t$ が自然にやっていた「過去の圧縮」を、外部ロジックで模倣    |
| **第4章** RAG・GraphRAG                                    | RNNにはなかった**外部記憶**の導入。隠れ状態に詰め込めない大量知識への対処     |
| **第5章** MemGPT（OS型メモリ管理）                         | RNNの固定サイズ隠れ状態 → 可変サイズの外部メモリへの進化                      |
| **第6章** Reflection（自己内省）                           | RNNのBPTT（誤差逆伝播）は暗黙的学習。Reflectionは**明示的・言語的な自己修正** |

---

#### 4. 補足：RNN的な発想は「回帰」している

2024〜2025年に注目されている **State Space Model（SSM）**、特に **Mamba** は、Transformerの並列学習能力を維持しつつ、RNNのように**推論時には再帰的に状態を更新する**ハイブリッドアーキテクチャ。

$$h_t = \bar{A} \cdot h_{t-1} + \bar{B} \cdot x_t$$

この式はRNNの隠れ状態の更新式とほぼ同じ構造であり、**一度捨てたRNNの「メモリ」という発想が、形を変えて戻ってきている**。

---

#### まとめ

教科書全体が扱っている課題 ── **ステートレスなモデルにいかにメモリを持たせるか** ── は、RNNが最初から持っていた性質をTransformerの枠組みで再発明する営みである。RNNを理解した上でこの教科書を読むと、各技術の必要性がより深く理解できる。

## 2: 2026/03/18 08:00

> **論文:** 『From Human Memory to AI Memory: A Survey on Memory Mechanisms in the Era of LLMs』(arXiv:2504.15965)

### 論文がやっていること（ざっくり全体像）

この論文は、LLMエージェントまわりで乱立している「メモリ機構」を整理するために、

- 人間の記憶の分類（短期／長期、エピソード／意味、手続き記憶など）
- LLMエージェントの記憶（コンテキストウィンドウ、RAG、ユーザプロファイル、KVキャッシュ、モデルパラメータ etc.）

を対応付けて、一つの**三次元（3D）×8区分（8 Quadrants）**の枠組みに落とし込む「見取り図」を提案しているサーベイです。
単に事例を列挙するのではなく、「記憶をどういう観点で分解すべきか」を与えてくれるのが特徴です。

---

### 中心となるアイデア：3つの軸と「3D-8Qタクソノミー」

論文のコアは、AIメモリを次の3軸で整理した 3D-8Q メモリ分類 です。

1. **Object（対象）**
   - Personal Memory：ユーザ固有の情報・履歴・好みなど
   - System Memory：タスク途中の中間結果やスキル、ワークフローなど、システム内部の記憶
2. **Form（形態）**
   - Non-Parametric：外部に保存されたメモリ（DB, ベクターストア, グラフ, ドキュメントなど）
   - Parametric：モデルパラメータとして埋め込まれたメモリ（事前学習知識や微調整で埋め込んだ個人情報など）
3. **Time（時間スケール）**
   - Short-Term：セッション内／その場限りの短期的な文脈（コンテキスト）
   - Long-Term：セッションをまたいで保持される長期メモリ

この 2×2×2 の組み合わせで 8つの象限（Quadrant I〜VIII） を定義し、それぞれにどんなメモリや研究が当てはまるかを整理します。
論文中の Table 1 が、この「8マスの見取り図」です。

---

### 人間の記憶モデルとの対応づけ

まず人間の記憶をおさらいし、それを LLM メモリに対応づけています。

- 短期記憶：感覚記憶＋ワーキングメモリ
- 長期記憶：
  - 明示的記憶（エピソード記憶・意味記憶）
  - 暗黙的記憶（手続き記憶）

これに対して、LLM 側では：

- **感覚記憶** → テキスト・画像・音声などの入力をトークン等の内部表現に変換する段階
- **ワーキングメモリ** →
  - 会話コンテキスト（ユーザとのマルチターン履歴）
  - Chain-of-Thought や ReAct で生成される思考／行動の中間結果
  - KV キャッシュなど推論時の一時状態
- **明示的記憶** →
  - エピソード記憶：ユーザごとの履歴・プロフィールを外部ストアに保存（非パラメトリック長期）
  - 意味記憶：モデルパラメータに埋め込まれた世界知識（パラメトリック長期）
- **暗黙的（手続き）記憶** →
  - エージェントが学習したスキル・手順（コード生成パターン、タスク分解の仕方など）
  - それが外部スキルライブラリやモデル内部のパターンとして蓄積される

つまり、「人間の短期／長期・エピソード／意味・手続き記憶」を、LLM エージェントの個人／システム・パラメトリック／非パラメトリック・短期／長期メモリにマッピングして、両者の構造の似方／違いを整理している、という位置づけです。

雰囲気だけ掴めるように、少し粗めにまとめます：

| Object   | Form           | Time       | Quadrant | 役割                 | 説明                                                                                                                                                                                       |
| :------- | :------------- | :--------- | :------: | :------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| Personal | Parametric     | Short-Term |   III    | ワーキングメモリ     | プロンプトキャッシュなど、特定ユーザのコンテキストをキャッシュして高速化する仕組み。そのユーザ向けの処理を一時的にパラメトリックに持っておくイメージ。                                     |

| System   | Parametric     | Long-Term  |   VIII   | 意味記憶＋手続き記憶 | 事前学習済み LLM 自体のパラメータに埋め込まれた知識・スキル、および Knowledge Editing/MemoryLLM/WISE などによる「継続的な知識更新」。世界知識・タスク知識を蓄積した長期記憶。              |

こうして見ると、同じ「Memory」と言っても、「ユーザの履歴を覚えるもの」から「KV キャッシュの圧縮」まで、かなり性質が違うものを一つの枠組みで見通しよく整理できる、というのがこのタクソノミーの狙いです。

論文本体は、3D-8Q の枠組みを提示したあと、

  - セッション内コンテキスト（I）

### 教科書の4指標を超える、LLMメモリシステムの評価指標群

#### 結論

教科書1.2節の4指標（精度・レイテンシ・コスト・情報の鮮度）は**「システム効率の基本軸」**に過ぎない。2024〜2026年の最新論文では**26以上の追加指標**が提案されており、大きく3方向（認知的忠実性・エージェント実用性・運用信頼性）に拡張されている。

---

#### 1. 教科書の4指標の位置づけ（復習）

| 指標                       | 焦点                   |
| :------------------------- | :--------------------- |
| **精度 (Accuracy)**        | 検索・回答の正確性     |
| **レイテンシ (Latency)**   | 応答速度               |
| **コスト (Cost)**          | トークン消費・計算資源 |

##### 2-6. スケーラビリティ

**3次元スケーラビリティ**（arXiv:2602.19320）：

| 次元                                       | 測定対象                   |
| :----------------------------------------- | :------------------------- |
| **Volume（量）**                           | 総トークン負荷の耐性       |
| **Interaction Depth（対話深度）**          | セッション横断の時間的構造 |
| **Entity Diversity（エンティティ多様性）** | 同時追跡する概念スレッド数 |

**MQI（メモリ品質指数）** — 精度・効率・容量を統合した複合指標：

$$MQI = \omega_1 \times Accuracy + \omega_2 \times Efficiency + \omega_3 \times Capacity$$

ここで $Efficiency = 1/(latency \times cost)$、$Capacity = max\_memory\_size \times retention\_rate$

### Transformerアーキテクチャにおける中核的なメカニズムであるSelf-Attention

生成AIにとっての「短期記憶」は、**コンテキストウィンドウ**と呼ばれます。これは、AIが一度に机の上に広げて見ることができる「資料の量」のようなものです。

#### Self-Attention（自己注意機構）とは
文章を読むとき、AIは「どの単語が、どの単語と関係が深いか」を計算します。これが**Self-Attention**です。

* **問題点：** 全ての単語（$N$個）が、自分自身を含む全ての単語（$N$個）との関係性をチェックします。
* **計算量 $O(N^2)$：** 単語の数が $N$ なら、計算量は $N \times N = N^2$ になります。
    * 単語が **2倍** になると、計算量とメモリ消費は **4倍**。
    * 単語が **10倍** になると、なんと **100倍** に膨れ上がります！

#### Attention と Feed-Forward（AIの2大エンジン）

Transformerというモデルは、基本的にこの2つの計算を交互に繰り返して動いています。

* **Attention（自己注意機構）**：
    * **役割**：単語と単語の「関係性」を調べるステップ。
    * **例え**：教室にいる全員を見渡して、「誰が誰と会話しているか」をチェックする作業。
    * **課題**：人数（系列長 $N$）が増えると、チェックすべきペアが $N^2$ で増えるため、ここでメモリがパンクします。

* **Feed-Forward（順伝播型ネットワーク）**：
    * **役割**：個々の単語の「意味」を深く解釈するステップ。
    * **例え**：Attentionで得た情報を元に、一人一人が自分のノートに「今の会話の意味」を整理して書き込む作業。
    * **特徴**：これは単語ごとに独立して計算できるため、Attentionほどメモリ問題は深刻ではありません。


### 2. Blockwise Parallel Transformers (BPT)

「巨大な表を一度に作るとメモリが足りないなら、分割して計算すればいい」という発想が**BPT**です。

* **仕組み：** 長い文章をいくつかの「ブロック」に切り分けます。
* **メリット：** 巨大なメモリを一度に消費せず、小さなメモリで計算を繰り返す（ループ処理）ことで、メモリ消費量を抑えます。
* **弱点：** 計算は効率化できましたが、複数の計算機（GPU）で分担しようとすると、データの送り迎え（通信）に時間がかかりすぎてしまうという課題が残りました。

#### $2bsh$ バイトの意味

これは「AIが計算中に、一時的にどれくらいのデータを手元に置いておく必要があるか」を示す体積のようなものです。

$$V = b \times s \times h \times 2$$

* **$b$ (Batch size)**：一度に処理する「文章の数」。（例：32本のニュース記事）
* **$s$ (Sequence length)**：1つの文章の「長さ（単語数）」。
* **$h$ (Hidden dimension)**：1つの単語を「どれくらいの細かさ（次元数）」で表現するか。
* **$2$ (bytes)**：半精度浮動小数点（16bit = 2byte）で計算することが多いため。

**なぜ重要か？**：
通常、Attentionの計算には $s^2$（系列長の2乗）のメモリが必要ですが、BPT（Blockwise Parallel Transformers）という手法を使うと、この **$s^2$ の部分を $s$（線形）のオーダーまで落とし込める** ことを示しています。

### 3. Ring Attention（最強のバケツリレー）

BPTの弱点を克服し、1億トークン（本数千冊分）という「ほぼ無限」の記憶容量を実現したのが**Ring Attention**です。

#### 仕組み：論理的なリング構造
複数のGPU（計算機）を円形（リング）につなぎます。

1.  **分担：** 文章を細切れにして、各GPUに持たせます。
2.  **バケツリレー（オーバーラップ通信）：** * GPU Aが自分の担当分を計算している間に、裏側でこっそり隣のGPU Bにデータを渡します。
    * 「計算」と「通信」を同時に行うことで、待ち時間をゼロにします。
3.  **線形スケール：** GPUを増やせば増やすほど、扱える文章の長さもそのまま増えていきます。

#### 「メモリ制約を完全に排除」の正体

「本当にそんな魔法があるの？」と疑うのは非常に正しい感覚です。ここでの「完全に排除」とは、正確には**「1台のデバイス（GPU）のメモリ容量が、処理できる文章の長さの限界を決めなくなった」**という意味です。

### 従来の限界
以前は、どんなにスーパーコンピューターを使っても、**「1つの計算（Attention行列）」は1枚のGPUチップの中で完結させる必要**がありました。そのため、チップのメモリ（例：80GB）がいっぱいになった時点で、それ以上長い文章は扱えませんでした。

### Ring Attention による突破
Ring Attentionは、巨大な計算を「輪（リング）」状に並んだ複数のGPUにバラバラに分散させます。

1.  **データの細分化**：1,000万トークンの文章を、10枚のGPUに100万トークンずつ持たせます。
2.  **計算のバケツリレー**：
    * GPU①は、自分が持つ「100万トークン分」の計算をします。
    * 終わったら、そのデータを隣のGPU②に渡します。
    * **ここが肝！**：GPU①はデータを渡したら忘れていい（メモリを空けられる）のです。
3.  **計算と通信の同時進行**：
    隣からデータが届くのを待つのではなく、**「今あるデータで計算している間に、次のデータを裏で受け取る」**という処理（オーバーラップ通信）を行います。


### なぜ「無限」と言えるのか？
この方式だと、**「文章を長くしたいなら、GPUを1枚追加すればいいだけ」**になります。
* 1枚のGPUのメモリが1GBしかなくても、1,000枚つなげば1,000GB分の文章が扱える。
* **「1枚の限界 ＝ 全体の限界」という呪縛が解けた**ため、理論上はデバイスを増やし続ける限り、コンテキスト（記憶容量）をどこまでも伸ばせるようになったのです。

### Softmax正規化の数学的特性と「ゼロサムの注意配分」

#### 1. Softmax関数とは何か（前提知識）

Self-Attentionでは、あるトークン（Query）が他のすべてのトークン（Key）とどれだけ「関連しているか」を数値化します。この生のスコア（logit）を、**確率分布に変換する**のがSoftmax関数です。

$$\alpha_{ij} = \text{softmax}(s_{ij}) = \frac{\exp(s_{ij})}{\sum_{l=1}^{N} \exp(s_{il})}$$

- $s_{ij}$：QueryトークンiとKeyトークンjの類似度スコア（内積）
- $N$：入力系列の全トークン数
- $\alpha_{ij}$：トークンiがトークンjに割り当てる**Attention重み**

#### 2. 「合計が必ず1になる」という制約

Softmax関数の最も重要な数学的特性は、出力が以下の2条件を**常に満たす**ことです。

| 条件       | 数式                             | 意味                                        |
| :--------- | :------------------------------- | :------------------------------------------ |
| **非負性** | $\alpha_{ij} \geq 0$             | 注意の重みは負にならない                    |
| **正規化** | $\sum_{j=1}^{N} \alpha_{ij} = 1$ | あるQueryから見た全Keyへの重みの合計は必ず1 |

つまり、Attention重みは**確率分布**として振る舞います。これは「注意という資源の総量が1に固定されている」ことを意味します。

#### 3. なぜこれが「ゼロサム」になるのか

教科書が「ゼロサムの構造」と呼んでいるのは、以下のメカニズムです。

**トークン数が少ない場合（N=3）:**

| Key           | スコア | Attention重み |
| :------------ | :----- | :------------ |
| A（関連あり） | 5.0    | **0.88**      |
| B             | 1.0    | 0.06          |
| C             | 1.0    | 0.06          |

→ 関連トークンAに注意の88%が集中。

**無関係なトークンを7個追加した場合（N=10）:**

分母の $\sum \exp(s_{il})$ に7個分の $\exp$ 値が加算されるため、**分母が大きくなり**、すべての $\alpha_{ij}$ が縮小します。

| Key            | スコア | Attention重み                |
| :------------- | :----- | :--------------------------- |
| A（関連あり）  | 5.0    | **0.65** （↓23ポイント低下） |
| B〜J（無関係） | 各1.0  | 各0.039                      |

これが教科書の記述する「**トークンが追加されるたびに、既存のすべてのトークンに割り当てられるAttentionの総量が減少する**」という現象の数学的実態です。パイ（=1.0）の大きさは変わらないのに、分け合う人数だけが増えていくイメージです。

#### 4. Attention Sinks（注意の掃き溜め）

ここからが教科書の核心部分です。

Softmaxは**「どこにも注意を向けない」という選択肢を許さない**関数です。MITの研究チームはこれを「**棄権が許されない民主主義投票（deafening democracy where abstention is disallowed）**」と表現しています。

すべての重みが非負かつ合計1であるため、仮にあるAttentionヘッドが「このクエリに対して本当に関連するトークンはない」と判断したくても、**どこかに注意を配分しなければなりません**。

この「行き場のない注意」が投棄される先が、入力の**先頭トークン**（BOS記号、システムプロンプトの冒頭、区切り文字など）です。これが**Attention Sinks**と呼ばれる現象です。

```text
[先頭トークン] ← 余剰Attentionが集中（Sink）
[トークン2]
[トークン3]    ← 本来注目すべき情報
...
[トークンN]
```

ICLR 2025で発表された研究によれば、Attention Sinksは深いモデルにおいて**表現の崩壊（representational collapse）を防ぐ安全弁**としても機能しており、単なるバグではなくSoftmaxベースのAttentionに内在する構造的な性質であることが明らかになっています。

#### 5. まとめ：Context Dilutionへの接続

| 特性             | 内容                                                                  |
| :--------------- | :-------------------------------------------------------------------- |
| **合計1の制約**  | 注意の総量は固定。トークンが増えれば1人あたりの取り分が減る           |
| **非負性の制約** | 「注意を向けない」という選択ができない                                |
| **結果**         | 無関係な情報を追加するほど、関連情報へのAttentionが**単調に減少**する |

この数学的特性こそが、教科書3.2.2以降で解説される「正解が見えているのに答えられない」現象（Context Dilution）の根本原因です。RAGの設計において「関連度の低い文書をプロンプトに混入させない」ことが極めて重要である理由は、このSoftmaxの数学的制約に直結しています。

> Replace, Don’t Expand: Mitigating Context Dilution in Multi-Hop RAG via Fixed-Budget Evidence Assembly

論文の主張の流れに沿って、大きなストーリーから順にまとめます。

### 1. 問題提起：マルチホップRAGの本当の敵は「足りない文書」ではなく「薄まる文脈」

著者たちはまず、「RAGはマルチホップ質問に弱い」というよく知られた課題から話を始めています。
従来は「最初の top‑k で橋渡し（bridge）情報を取り逃がすと詰む」と言われ、それを防ぐために Self‑RAG や CRAG のような反復・自己修正型RAGが提案されてきました。

しかしこれらの多くは「怪しいと思ったら文書をどんどん足していく」戦略（Breadth‑First Addition）を取ります。
その結果、現実の制約（レイテンシ・トークン上限）がある環境では、

- 関係ない文書（distractors）が大量に混ざる
- 同じ内容の横方向重複（lateral redundancy）が増える
- モデルが必要な橋渡し情報を拾えず、「文脈が薄まる（context dilution）」

という現象が起き、かえって推論性能が落ちると指摘しています。
ここで著者は「少ない予算（固定 `k`）の中でどう文書セットを最適化するか」という観点に問題を切り替えます。

### 2. 提案：増やすな、入れ替えろ ― 固定予算の Evidence Assembly としてのSEAL‑RAG

そこで導入されるのが SEAL‑RAG です。
このシステムは、RAGを「どんどん証拠を溜め込むプロセス」ではなく、サイズ `k` が固定された証拠集合 `E` を最適化するセット最適化問題として捉え直します。

主張の核は、

- コンテキストスロット `k` は「余ったら詰め込むバッファ」ではなく、極めて貴重なリソース

- だからやるべきことは「追加」ではなく、低価値な文書を追い出して高価値な文書に差し替える（Replace, Don’t Expand）

- その差し替えは「なんとなくの関連度」ではなく、明示的に特定した“欠けている情報（ギャップ）”をどれだけ埋めるかで評価するべき

というものです。

この思想を具体化したのが、論文タイトルにもある「Fixed‑Budget Evidence Assembly（固定予算の証拠組み立て）」であり、それを制御するコントローラが SEAL‑RAG です。

### 3. コアの仕組み：Search → Extract → Assess → Loop（SEAL）サイクル

SEAL‑RAGは、次のループ構造のコントローラとして定義されています。

1. **Search（検索）**
   通常の dense/hybrid retriever で、まず top‑k の文書集合 `E₀` を取ってくる（ここで `k` は固定。その後も常に `|Eₜ| = k` を維持）。

2. **Extract（抽出）**
   この `Eₜ` から、OpenIE ベースの軽量な抽出モジュールで
   - エンティティとエイリアス
   - 関係 `(h, r, t)`
   - 日付・場所などの修飾子（qualifier）

   を取り出し、Entity Ledger `Uₜ` という構造化表現を作ります。
   抽出は必ずテキストに明示的に書いてあることだけ（verbatim constraint）に限定し、幻覚を避けます。

3. **Assess（評価／十分性判定）**
   質問 `q` と Ledger `Uₜ` に基づき、以下のシグナルを LLM でスコアリングして、
   - **Coverage**：質問に必要な属性がどれだけ埋まっているか
   - **Corroboration**：重要事実が複数ソースで裏付けられているか
   - **Contradiction**：矛盾する情報がないか
   - **Answerability**：今の情報で答えられそうか

   から、`Suff(q, Uₜ)`（今の証拠で十分か）を判定します。
   十分ならここでループ終了し、生成ステップへ進みます。

4. **Loop（修復ループ：ギャップ診断 → マイクロクエリ → 置き換え）**
   十分でない場合、SEAL‑RAGは「何が足りないのか」を明示的に計算します。
   - 質問 `q` から必要情報の集合 `N(q)` を構成し、Ledger `Uₜ` に既にあるものを差し引くことで Explicit Gap Specification `Gₜ = N(q) \ Uₜ` を作る
   - `Gₜ` は、以下に分類される
     1. Missing Entity（橋渡しエンティティがない）
     2. Missing Relation（エンティティはあるが関係がない）
     3. Missing Qualifier（日付など属性だけ足りない）
   - 各ギャップに対して「超ピンポイントなマイクロクエリ」（例：Blur の Parklife のリリース年）を作り、retriever を再実行して候補集合 `Cₜ` を得る
   - その候補 `c∈Cₜ` について、Entity‑First Utility `S(c)` を計算し、`Eₜ` 内の「いちばん役に立っていない文書」と比較して、より有用なら入れ替える

この Search → Extract → Assess → Loop を、ループ予算 `L` まで繰り返しますが、どれだけループしても文書数は常に `k` のままです。

### 4. 「置き換え」を成り立たせる技術：Entity‑First Utility と入れ替えポリシー

置き換えの鍵は、「どの候補を残して、どれを追い出すか」をどう決めるかです。
著者は、MMR に着想を得た Entity‑First Utility 関数

`S(c) = λ₁ GapCov + λ₂ Corr + λ₃ Nov − λ₄ Red`

を導入し、候補文書 `c` の価値を次の観点で評価します。

- **GapCov**：ギャップ集合 `Gₜ` に書かれている「足りていないエンティティ／関係」を含んでいるか
- **Corr**：既存の Ledger `Uₜ` の事実を裏付けるか（信頼度アップ）
- **Nov**：まだ出ていない新しいエンティティ／関係を持っているか（非冗長な新情報）
- **Red**：既存 `Eₜ` とどれくらい重複しているか（高いとマイナス）

このスコアに基づき、

- `Eₜ` の中で最もスコアの低い文書 `v`
- `Cₜ` の中で最もスコアの高い候補 `c*`

を比べて、`S(c*) > S(v) + ε` なら `v` を追い出して `c*` を入れる、というルールで更新します。
さらに、

- 新規に入れた文書は1ステップは追い出し対象から守る（Dwell‑Time Guard）
- `ε` のしきい値で「ほとんど変わらない入れ替え」を抑制（スラッシング防止）

といった工夫も入れて、トークン数を増やさずに情報密度を上げることを狙っています。

### 5. 計算コストの主張：推論コストが事前に読める「固定予算」のメリット

次に著者は、「この設計は現実のシステム運用に向いている」と主張します。
SEAL‑RAGは、

- 検索＋抽出＋評価のループは最大 `L` 回
- 生成は最後に1回だけ、コンテキストサイズは常に `k`

なので、コストは概ね

- 検索＋抽出：`O(L)`
- 生成：`O(k)`（一度だけ）

となり、`O(k·L)` の一定枠に収まると議論します。
一方、Addベースの方法はループするたびにコンテキストが増え、生成コストがループ数に応じて膨らんでいきます。
ここでも「Replace, Don’t Expand」は「推論コストも読める」という実務上の利点を持つ、と位置づけられています。

### 6. 実験：SEAL‑RAGは本当に「増やすより入れ替えたほうが強い」のか

ここからは、この設計が本当に効くかを HotpotQA と 2WikiMultiHopQA で検証した結果の話です。
全ての手法は LangGraph 上に統一環境で実装され、

- 同じ index（Wikipediaをページ単位でベクトル化）
- 同じ retriever（OpenAI embedding＋Pinecone）
- 同じ LLM バックエンド（GPT‑4 系）
- 同じ Judge‑EM 評価（GPT‑4o をジャッジとして利用）

という条件で比較されています。

比較対象は、

- Basic RAG（単純な Retrieve→Generate）
- Self‑RAG（反省ループでクエリ変形）
- CRAG（「ダメならWeb検索を追加する」型の補正）
- Adaptive‑k（最初に大量に取ってきて、類似度の”ギャップ”で `k` を動的に決める選択型）

です。

#### 6.1 k = 1：1スロットしかない極限条件での検証

`k=1` の場合、証拠スロットは「1つだけ」です。
ここでは「その1枠に橋渡し文書を置けるか」がすべてになります。

結果として、

- SEAL‑RAG は全てのモデルで Judge‑EM を +7〜+19ポイント ベースラインより改善
- Precision は維持／向上しつつ、Recall も同等以上を確保
- Self‑RAG や CRAG などの Add型は、追加した文書が `k>1` の位置に押し出されてしまい、最終的に切り捨てられることが多い

という形で、「`k=1` では置き換え能力を持つかどうかが致命的に効く」と示しています。

ここで著者は面白い解釈をしていて、

- 通常のRAG：複数文書を同時に見て推論する「Read‑Time Reasoning」を要求するが、`k=1` ではそもそも不可能
- SEAL‑RAG：橋渡しなど一部の推論をRetrieval‑Time Reasoning（Ledger上で段階的に解決）にオフロードすることで、`k=1` でもマルチホップを解ける

と位置づけています。

#### 6.2 k = 3：実務的な設定での Precision vs Recall

`k=3` は、実務でもよく使われるスロット数です。
ここでは「ある程度余裕のあるスロットをどう埋めるか」が問われます。

- 全体的に、どの手法も Recall は高くなる
- しかし、SEAL‑RAG は Precision が圧倒的に高い（例：gpt‑4o で 89% vs 他は最大でも 76%）
- Add型の手法は、答えを含む文書は拾えても、それ以外をノイズで埋めてしまい、最終的な答えの正しさ（Judge‑EM）が落ちる

という結果から、「`k` を増やしても、入れ替えで冗長性を削ぎ落とさないとノイズに負ける」という主張を補強しています。

#### 6.3 2WikiMultiHopQA ＆ k = 5：’More Context is Better’ 仮説の崩壊

より複雑なマルチホップ推論を要求する 2WikiMultiHopQA で、`k=1,3,5` を評価したところ、

- `k=1` でも SEAL‑RAG は 61〜76% の正解率を出し、基本的な手法が 14〜30% にとどまる中で大幅に上回る
- `k=5` にすると、CRAG や Basic RAG の Precision は 11〜34% まで崩壊（文書はたくさんあるが、ほとんどノイズ）
- 一方、SEAL‑RAG は `k=5` でも Precision 89〜96% を維持

となり、「コンテキストを増やすとむしろ薄まる」という Context Dilution の実測を示しつつ、「Replace戦略なら `k` を増やしても精度を保てる」ことを示しています。

### 7. Adaptive‑k との直接対決：選択（Selection）だけでは足りない

次に著者は、最近の「たくさん取ってから賢く削る」系である Adaptive‑k と、同じ 2Wiki `k=5` 設定で比較します。

- **Adaptive‑k (No Buffer)**：
  - 初期 pool から”最大ギャップ”位置で切る
  - Precision は高い（86%）が、肝心の橋渡し文書が最初の pool にないとどうにもならない → 正解率は 40〜41%
- **Adaptive‑k (Buffer)**：
  - Recall を上げようと”余裕（バッファ）”を持たせる
  - 正解率は上がるが、Precision が 26% まで落ちる（ノイズだらけ）

これに対し、SEAL‑RAG は、

- 初期の pool にない情報も、マイクロクエリで新規に取りに行く（Active Repair）
- そのうえで、GapCov と Novelty を見ながら不要文書を追い出していく

という動的修復の仕組みによって、

- Adaptive‑k より +7.5〜+8.0 ポイントの正解率向上
- Precision も 89〜96% と高いまま

を達成します。
ここから、「最初の候補集合の中だけでやりくりする **選択型（selector）では限界があり、マルチホップでは新しい証拠を能動的に取りに行く修復型（repairer）** が必要だ」という主張が導かれています。

### 8. 追加分析：ループ数・モジュールごとの貢献・失敗パターン

#### 8.1 ループ予算 L の効果

`k=1`, HotpotQA でループ回数 `L` を 0,1,3,5 と変えた結果、

- `L=0`（ループなし）から `L=1` にするだけで、平均で +35ポイント も Judge‑EM が上がる
- それ以降（`L=3,5`）は漸近的な改善で、効果は小さい

つまり、「1回目の修復」だけでほとんどの利得が得られることがわかり、「深いエージェントループではなく、明示的なギャップ駆動の一撃で多くのケースが解決する」と主張しています。

#### 8.2 モジュールごとの役割

著者は、成功例のトレースから次の3点を強調します。

- 単なるキーワード検索ではなく、Ledger に基づく構造化のギャップ検出が、日付などの細かい属性取得に効いている
- Self‑Ask 的な広いクエリ分解ではなく、1ギャップ＝1マイクロクエリの粒度に抑えることで、ノイズだらけの文書を避けている
- Relevance のみのランキングではなく、Novelty と Redundancy の調整で、意味のない重複文書をちゃんと捨てている

と述べ、SEAL‑RAGの性能は単一のテクニックではなく、これらの組み合わせに依存しているとまとめています。

#### 8.3 失敗例と限界

失敗パターンとしては、

- エイリアス解決の失敗（Apple Inc. vs Apple Computer など）
- 抽出モジュールのノイズや取りこぼし（複雑な文章構造で qualifier を見逃す）

などがあり、将来的にはエンティティリンクや検証ステップの強化で改善できそうだが、レイテンシとのトレードオフになると指摘しています。

### 9. 結論：固定予算下では「証拠を増やす」より「証拠集合を最適化する」べき

最後に著者は、この論文のメッセージを次のように総括しています。

- マルチホップRAGにおいて、真のボトルネックは「十分な recall がないこと」だけではなく、増えすぎたコンテキストによる Context Dilution である
- そのため、証拠集合 `E` のサイズ `k` を固定したうえで、その中身をどれだけギャップ充足的・高密度に最適化するかという視点が重要
- SEAL‑RAGは、そのための具体的なコントローラ設計（Explicit Gap Specification ＋ Entity‑First Replacement）を示し、HotpotQA / 2WikiMultiHopQA の両方で Add型／Prune型の先行手法を上回ることを示した
- 特に `k=5` など、従来は「多く取れば有利」と思われがちな設定でも、Add型は Precision 崩壊を起こす一方で、SEAL‑RAGは Precision を 96% まで保てる

そして今後の方向性としては、

- ループ制御の軽量化（小さいモデルによる sufficiency 判定の蒸留など）
- 法律・医療など高精度が必須な領域への応用

を挙げつつ、「固定予算の evidence assembly は、実運用のRAGにとって有望なパラダイムである」と締めくくっています。

もし「実装寄りの観点で、既存のRAGパイプラインにSEAL的な考え方をどう取り入れるか」を整理したくなったら、その視点でも整理できます。例えば、既にあるRAGに「ギャップ検出＋マイクロクエリ＋入れ替えランキング」だけを足す、などです。

### Anthropic「Contextual Retrieval」の実装と、ハイブリッド検索アーキテクチャの全体像

---

### 1. Contextual Retrieval とは何か

#### 1.1 解決する課題

従来のRAG（Naive RAG）では、ドキュメントをチャンク分割した時点で**「そのチャンクが文書全体の中でどういう位置づけにあるか」**という情報が失われる。これが教科書4.1節で述べられた「コンテキストの孤児（Contextual orphans）」問題である。

例えば、以下のチャンクだけでは主語も時期も不明である：

> `"The company's revenue grew by 3% over the previous quarter."`

#### 1.2 核心的なアイデア

Contextual Retrievalは、**各チャンクをembeddingする前に、LLMを使ってそのチャンクの文脈説明（50〜100トークン）を先頭に付加する**というシンプルな手法である。

|                        | 内容                                                                                                                                                                                     |
| :--------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **元のチャンク**       | `"The company's revenue grew by 3% over the previous quarter."`                                                                                                                          |
| **コンテキスト付加後** | `"This chunk is from an SEC filing on ACME corp's performance in Q2 2023; the previous quarter's revenue was $314 million. The company's revenue grew by 3% over the previous quarter."` |

「どの会社の、いつの、何の文脈か」が明示されることで、embeddingベクトルの意味的品質が向上し、検索精度が飛躍的に改善する。

---

### 2. 具体的な実装

#### 2.1 コンテキスト生成プロンプト（Anthropic公式）

Anthropicが公式ブログおよびCookbookで公開しているプロンプトテンプレート：

```text
<document>
{doc_content}
</document>
Here is the chunk we want to situate within the whole document
<chunk>
{chunk_content}
</chunk>

Please give a short succinct context to situate this chunk within the overall
document for the purposes of improving search retrieval of the chunk.
Answer only with the succinct context and nothing else.
```

- `{doc_content}` にドキュメント全文を、`{chunk_content}` に対象チャンクを渡す
- 出力は50〜100トークン程度の簡潔な文脈説明

#### 2.2 コンテキスト生成コード（Prompt Caching活用）

```python
import anthropic

DOCUMENT_CONTEXT_PROMPT = """<document>
{doc_content}
</document>
Here is the chunk we want to situate within the whole document
"""

CHUNK_CONTEXT_PROMPT = """<chunk>
{chunk_content}
</chunk>

Please give a short succinct context to situate this chunk within the overall
document for the purposes of improving search retrieval of the chunk.
Answer only with the succinct context and nothing else."""

def situate_context(client: anthropic.Anthropic, doc: str, chunk: str) -> str:
    response = client.messages.create(
        model="claude-haiku-4-5",
        max_tokens=1000,
        temperature=0.0,
        messages=[{
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": DOCUMENT_CONTEXT_PROMPT.format(doc_content=doc),
                    "cache_control": {"type": "ephemeral"},  # ← ドキュメント部分をキャッシュ
                },
                {
                    "type": "text",
                    "text": CHUNK_CONTEXT_PROMPT.format(chunk_content=chunk),
                },
            ],
        }],
    )
    return response.content[0].text
```

**`cache_control` が鍵：** 同一ドキュメントの複数チャンクを処理する際、ドキュメント全文部分のKVキャッシュが再利用される。これにより入力トークンの70〜80%がキャッシュから読み取られ、**90%割引**が適用される。

#### 2.3 チャンクの結合とインデックス登録

```python
def process_document(client, doc_content: str, chunks: list[str]):
    processed = []
    for chunk in chunks:
        context = situate_context(client, doc_content, chunk)
        processed.append({
            # コンテキスト + 元チャンクを結合してembedding
            "text_to_embed": f"{context}\n\n{chunk}",
            "original_content": chunk,            # ← LLMに渡す用（元テキスト）
            "contextualized_content": context,    # ← 生成した文脈
        })
    return processed
```

生成後のデータは以下のように保存する：

```python
record = {
    "id": "chunk_042",
    "vector": embed(text_to_embed),          # ← 検索用（コンテキスト込み）
    "metadata": {
        "original_content": chunk,            # ← LLMに渡す用（元テキスト）
        "contextualized_content": context,    # ← 生成した文脈
        "doc_id": "sec_filing_2023_q2",
        "page": 12,
    }
}
vector_db.upsert(record)
```

**設計上の重要ポイント：** `text_to_embed`（コンテキスト込み）は**検索精度向上のため**にembeddingに使い、`original_content`（元チャンク）は**LLMへの入力用**として別途メタデータに保存する。検索とLLM入力で使うテキストを分離することで、検索精度の向上とLLMへのトークン節約を両立できる。

#### 2.4 コスト目安

Prompt Caching適用時、**100万ドキュメントトークンあたり約$1.02**（800トークンチャンク、8Kドキュメント前提）。キャッシュヒット分は90%割引が適用されるため、ナイーブにHaikuを呼ぶよりも大幅に安価。

---

### 3. コンテキスト付加によるインデックスへの影響

「チャンクに文脈を付加してembeddingすると、インデックスの量やembeddingモデルの入力トークン量が膨大になるのではないか」という疑問が生じるが、結論として**インデックスサイズはほぼ変わらず、入力トークン量の増加も軽微**である。

#### 3.1 embeddingの出力は固定長ベクトル

embeddingモデルは入力テキストの長さに**関係なく**、常に**固定次元のベクトル**を出力する。

```text
"revenue grew by 3%"                          → [0.12, -0.34, ..., 0.56]  (1024次元)
"SEC filing Q2 2023, ACME. revenue grew by 3%" → [0.18, -0.29, ..., 0.61]  (1024次元)
                                                  ↑ 同じサイズ
```

| 項目                              | 通常チャンク       | コンテキスト付きチャンク | 差分         |
| :-------------------------------- | :----------------- | :----------------------- | :----------- |
| embeddingへの入力                 | ~512トークン       | ~562〜612トークン        | **+10〜20%** |
| **Vector DBのインデックスサイズ** | 1024次元 × float32 | 1024次元 × float32       | **変化なし** |
| BM25インデックス                  | 512トークン分      | 612トークン分            | 微増         |

つまり、**ベクトル検索のインデックス容量は1バイトも増えない**。増えるのはembeddingモデルへの入力トークンだけで、それも1チャンクあたり50〜100トークン（約10〜20%増）に過ぎない。

#### 3.2 embeddingモデルの入力上限も問題なし

主要なembeddingモデルの最大入力トークン数：

| モデル                            | 最大入力トークン |
| :-------------------------------- | :--------------- |
| Voyage AI (`voyage-2`)            | **16,000**       |
| OpenAI (`text-embedding-3-small`) | **8,191**        |
| Cohere (`embed-v4`)               | **128,000**      |

通常のチャンクサイズが512〜1024トークンなので、50〜100トークンの文脈を付加しても**上限の数%にしかならない**。

#### 3.3 コンテキストの位置づけ

コンテキストは**「検索精度を上げるためのメタデータ」**と捉えると整理しやすい。embeddingに焼き込まれた後は、ベクトルの意味的品質として反映されるだけで、ストレージを圧迫する要因にはならない。

---

### 4. BM25とは何か

#### 4.1 定義

**BM25（Best Matching 25）**は、「キーワードの出現頻度」に基づいてドキュメントをランク付けする**古典的な全文検索アルゴリズム**である。Elasticsearchなどの検索エンジンのデフォルトスコアリングに採用されている。

#### 4.2 スコアリングの仕組み

BM25は、各クエリ単語について以下の2つの要素でスコアを計算する。

| 要素                                  | 考え方                                                                                                                     |
| :------------------------------------ | :------------------------------------------------------------------------------------------------------------------------- |
| **TF（Term Frequency）**              | そのチャンク内にクエリ単語が**多く出現**するほどスコアが高い。ただし飽和関数により、一定以上は頭打ちになる                 |
| **IDF（Inverse Document Frequency）** | コーパス全体で**珍しい単語**ほど重要視される。「の」「は」のような頻出語は低スコア、「GraphRAG」のような専門用語は高スコア |

```text
BM25("GraphRAG の実装方法")

  "GraphRAG" → IDF高い（珍しい語）→ 重み大
  "の"       → IDF低い（どこにでもある）→ 重みほぼゼロ
  "実装"     → IDF中程度 → 重み中
  "方法"     → IDF中程度 → 重み中
```

---

### 5. Embedding + Vector DB と BM25 の本質的な違い

両者は**全く異なる原理**で「関連性」を判定する。

```text
クエリ: 「犬の散歩にかかる費用」

┌─────────────────────────────────────────────────────┐
│ BM25（語彙マッチング）                                │
│                                                       │
│  「犬」「散歩」「費用」という"単語そのもの"が           │
│   含まれるチャンクを探す                               │
│                                                       │
│  ✅ 「犬の散歩代行サービスの費用は…」  → ヒット        │
│  ❌ 「ペットの運動にかかるコスト」     → ヒットしない   │
│     （同じ意味だが単語が違う）                         │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ Embedding + Vector DB（意味マッチング）                │
│                                                       │
│  テキストを「意味の座標」に変換し、                     │
│  座標が近いチャンクを探す                              │
│                                                       │
│  ✅ 「犬の散歩代行サービスの費用は…」  → ヒット        │
│  ✅ 「ペットの運動にかかるコスト」     → ヒット        │
│     （意味的に近いのでベクトルも近い）                  │
│  ❌ 「犬も歩けば棒に当たる」           → ヒットしがち   │
│     （「犬」「歩」が意味空間で近い）                    │
└─────────────────────────────────────────────────────┘
```

|                  | **BM25**                              | **Embedding + Vector DB**           |
| :--------------- | :------------------------------------ | :---------------------------------- |
| **原理**         | 単語の一致・頻度                      | 意味の類似度（ベクトル間距離）      |
| **得意**         | 固有名詞、型番、ID、正確な用語        | 言い換え、類義語、抽象的な質問      |
| **苦手**         | 同義語・言い換えへの対応              | 固有名詞の正確な一致                |
| **インデックス** | 転置インデックス（単語→文書の対応表） | ベクトルインデックス（ANN検索）     |
| **代表的な実装** | Elasticsearch, OpenSearch             | Pinecone, Weaviate, Qdrant          |
| **前処理コスト** | 低い（トークン化のみ）                | 高い（embeddingモデルの推論が必要） |

#### なぜ「ハイブリッド」が有効なのか

どちらか一方では必ず取りこぼしが発生する。

```text
クエリ: 「エラーコード E-4021 の対処法」

BM25:  "E-4021" を正確にマッチ → ✅ ピンポイントで発見
Vector: "E-4021" はembedding空間で意味を持たない → ❌ 見逃す

クエリ: 「アプリが突然落ちる問題」

BM25:  「落ちる」「問題」では関連ドキュメントが多すぎてノイズだらけ → △
Vector: 「クラッシュ」「異常終了」も意味的に近い → ✅ 広く拾える
```

**両方を組み合わせることで、互いの弱点を補完する**のがハイブリッド検索の設計意図である。

### 6. ハイブリッド検索の実装

#### 6.1 Rank Fusion + Reranking コード

```python
import cohere

def retrieve(query: str, vector_db, bm25_index, co_client: cohere.Client, k=20):
    # 1. 両系統から候補を広めに取得
    semantic_results = vector_db.search(query, k=150)
    bm25_results = bm25_index.search(query, k=150)

    # 2. Weighted Reciprocal Rank Fusion
    scores = {}
    for i, r in enumerate(semantic_results):
        scores[r.id] = scores.get(r.id, 0) + 0.8 * (1 / (i + 1))
    for i, r in enumerate(bm25_results):
        scores[r.id] = scores.get(r.id, 0) + 0.2 * (1 / (i + 1))

    # 3. 上位候補をRerankingで精緻化
    top_candidates = sorted(scores, key=scores.get, reverse=True)[:150]
    documents = [get_chunk_text(cid) for cid in top_candidates]

    reranked = co_client.rerank(
        model="rerank-english-v3.0",
        query=query,
        documents=documents,
        top_n=k,
    )
    return [documents[r.index] for r in reranked.results]
```

#### 6.2 Rank Fusion の計算例

```text
チャンクA: Vector DB 2位、BM25 1位
  → 0.8 × (1/2) + 0.2 × (1/1) = 0.4 + 0.2 = 0.60

チャンクB: Vector DB 1位、BM25 圏外
  → 0.8 × (1/1) + 0.2 × 0     = 0.8 + 0   = 0.80

チャンクC: Vector DB 3位、BM25 2位
  → 0.8 × (1/3) + 0.2 × (1/2) = 0.27 + 0.1 = 0.37

→ ランキング: B (0.80) > A (0.60) > C (0.37)
```

意味的に非常に近い（Vector 1位）チャンクBが最上位になりつつ、キーワードも一致するチャンクAも高順位に残る、というバランスの取れた結果になる。

### 11. LLMへの入力時の設計判断

検索で取得したチャンクをLLMに渡す際、3つの方式が考えられる：

| 方式                        | LLMへの入力                 | メリット                 | デメリット           |
| :-------------------------- | :-------------------------- | :----------------------- | :------------------- |
| **元チャンクのみ**          | `original_content`          | トークン節約、ノイズ最小 | 文脈が欠ける場合あり |
| **コンテキスト+元チャンク** | `context + original`        | 文脈が保たれる           | トークン微増         |
| **元チャンク+メタデータ**   | `original` + `doc_id, page` | 出典明示                 | 実装がやや複雑       |

実用上は、**検索にはコンテキスト込みベクトルを使い、LLMへの入力は元チャンク（＋必要に応じてコンテキスト）を使う**のが一般的である。

### まとめ：Contextual Retrieval の設計思想

教科書3.2.5節の設計原則「コンテキストの純度を最大化せよ」を、Contextual Retrievalは以下の形で体現している：

1. **検索精度の向上**：チャンクに文脈を付加してembeddingの意味的品質を高め、検索段階でのノイズ混入を防ぐ
2. **ハイブリッド検索による補完**：意味検索（Vector DB）とキーワード検索（BM25）の組み合わせで、両者の弱点を相互に補完する
3. **Rerankingによる精緻化**：粗い検索結果をクロスエンコーダで再評価し、真に関連性の高いチャンクのみをLLMに渡す
4. **インデックスへの影響は最小限**：コンテキストはembeddingに焼き込まれるだけで、Vector DBのストレージサイズは変化しない

これは教科書3.2.4節で紹介されたSEAL-RAGの「Replace, Don't Expand」原則とも通底する思想であり、**LLMに渡す情報量を増やすのではなく、渡す情報の質を高める**ことが、Context Dilution問題への根本的な対処法である。

> 質問: Graph RAGとは？詳細なメカニズムは？
> 起点：教科書4.3章

グラフRAGは、一言でいうと「知識グラフで文書全体の構造を作ってからRAGする方式」です。
普通のRAGが「ベクトル類似度で"それっぽいチャンク"を拾ってくる検索エンジン」だとしたら、GraphRAGは「エンティティと関係で"世界モデル"を組み立ててから、そのグラフを辿って答える検索＋要約エンジン」というイメージに近いです。

### グラフRAGとは何か（概要）

大きな文書集合（数十万〜数百万トークン）に対して、

- 人・組織・場所・イベント・概念などのエンティティ
- エンティティ同士の関係（誰が誰を買収、どこで何が起きた…）
- それらが作るクラスタ（コミュニティ）とテーマ

をLLMで抽出して知識グラフを構築し、そのグラフを基盤にRAGを行う手法です。

これにより、

- 「Novorossiyaは何をしたか？」「このデータセット全体の主要テーマは？」のようなマルチホップ推論・コーパス全体のテーマ要約（query-focused summarization）が得意になる
- 単に"質問と似ているチャンク"ではなく、"関連エンティティやコミュニティ全体"から文脈を集められる
- 回答とともに、どのエンティティ・どの関係・どの原文に根拠があるかという**プロベナンス（出典の筋道）**を提示しやすい

という特徴があります。

### 全体アーキテクチャ：2フェーズ構成

GraphRAGのコアは、

1. **インデックス構築フェーズ（オフライン）**
2. **クエリ応答フェーズ（オンライン）**

に分かれます。

### 1. インデックス構築フェーズ（グラフを作る）

#### (1) 文書分割（チャンク化）

- 文書（PDF / レポート / メール等）を、段落や意味単位ごとのテキストユニットに分割
- 各ユニットに一意なIDを付与（例: `doc1_paragraph_3`）

これは通常のベクトルRAGと同じですが、「後でエンティティ・関係に紐づけるための最小単位」として使われます。

#### (2) エンティティ・関係・クレーム抽出（LLM）

各テキストユニットをLLMに投げて、プロンプトで構造化抽出します。

- **エンティティ**
  - 例：人物、組織、地名、プロダクト、イベント、抽象概念など
  - それぞれに説明文を付ける（例：「Federal Reserve: 米国の中央銀行システムで、金融政策を司る機関」）
- **関係（リレーション / エッジ）**
  - 「AがBを買収」「XがYに影響を与えた」「組織OがイベントEを主催」など
  - エッジの種類や向き、強さ（出現頻度など）をメタデータとして持つ
- **クレーム（重要な事実文）**
  - 「A社が2023年Q3にB社を買収した」など、あとで要約や説明に使える"事実候補"

Microsoftの実装ではプロンプト設計が重要で、2024/09のauto-tuningでは「コーパスを見て自動で最適な抽出プロンプトを生成」する機構も入っています。

#### (3) 知識グラフ構築

抽出結果をもとに、グラフ構造を作ります。

- **ノード（頂点）** ＝エンティティ
- **エッジ（辺）** ＝エンティティ間の関係
- 各ノードに：
  - 名前
  - 説明文
  - 出現したテキストユニットの参照
- 各エッジに：
  - 関係タイプ（acquired, located_in, part_of など）
  - 出現元テキスト
  - 重み（頻度や重要度）

バックエンドはNeo4j / MemgraphなどのグラフDBでも、パーケット＋インメモリでも構成できますが、スケールするとグラフDBがほぼ必須になります。

#### (4) コミュニティ検出（Leidenアルゴリズム）

生成したグラフに対して、Leidenアルゴリズムでコミュニティ（クラスター）を検出します。

- **目的：**
  - 強く関連するエンティティ群を「コミュニティ」としてまとめる
  - コミュニティ間の階層構造（ローカル→中間→グローバル）を作る
- **Leidenの特徴：**
  - 旧来のLouvain法よりも「内部がよくつながったコミュニティ」を保証
  - 階層的に：
    - レベル0：元のグラフ
    - レベル1：密につながったノード群
    - レベル2：レベル1のコミュニティをさらにまとめた大きなテーマ
    - …と抽象度を上げていく

結果として、

- 低レベル：具体的な人物・プロジェクト・イベント
- 中レベル：部署 / プロダクトライン / キャンペーンなど
- 高レベル：企業戦略 / 戦争全体のテーマ / 顧客満足の主要因

といった「多層的な構造化」が得られます。

#### (5) コミュニティ要約（LLM）

各コミュニティに対して、LLMで要約レポートを作ります。

- 何のコミュニティなのか（テーマ名・概要）
- 含まれている主なエンティティ（重要人物・組織・場所）
- 代表的な関係やイベント
- 重要なクレームや典型的な文脈
- 出典となるテキストユニットへの参照

これを全レベルのコミュニティに対して行うので、後で「データセット全体のテーマ」も「あるサブトピックの特徴」も、事前に要約済みのレポートを組み合わせて答えられるようになります。

※ここがフルGraphRAGでは最もコストの重い部分で、LazyGraphRAGではここを削って、クエリ時に必要分だけLLMを使うやり方に変えています。

#### (6) ベクトル化とストレージ

最後に、以下を埋め込みベクトルにしてベクトルDBに保存します。

- エンティティ説明
- コミュニティ要約レポート
- 元のテキストユニット自体（通常のRAG用）

GraphRAG 1.0 では、これらをParquetに直に埋めるのではなく、専用のベクトルストア（LanceDBやAzure AI Search）に分離し、ディスク使用量を大きく削減しています。

### 2. クエリ応答フェーズ（グラフを使って答える）

GraphRAGのクエリは、大きく「グローバル」「ローカル」「ハイブリッド（DRIFTなど）」に分かれます。

#### (1) グローバル検索（Global Search）

**対象：** 「このデータセット全体の主要テーマは？」「このニュースコーパスにおける上位5つの関心事は？」といったコーパス全体を俯瞰する質問。

**流れ：**

1. ユーザ質問を埋め込みベクトル化
2. 全レベルのコミュニティ要約レポートとの類似度を計算
3. 動的コミュニティ選択で、レベルを固定せず、最も関連の高いコミュニティを複数選ぶ
4. 選ばれた各コミュニティレポートをLLMに渡し、「そのコミュニティ視点の部分回答」を生成
5. 全ての部分回答を統合し、最終的なグローバル回答を生成

普通のベクトルRAGだと、「トップkチャンク」をベクトル類似度で取ってくるだけなので、データセット全体の構造・重要テーマは見えませんが、GraphRAGは「階層コミュニティ要約」を通じてこれを解決します。

#### (2) ローカル検索（Local Search）

**対象：** 「Novorossiyaは何をしたか？」「CompanyXとその周辺の関係を教えて」など、特定エンティティや局所構造を詳しく知りたい質問。

**流れ：**

1. 質問文から、エンティティ（Novorossiya 等）を特定
   - 埋め込み＋名前マッチでノードを同定（将来はもっと高度なエンティティリンク）
2. そのノード周辺の**近傍グラフ（k-hop近傍）**を抽出
   - 関連するエンティティ・関係・テキストユニットを集める
3. 近傍に関連するコミュニティ要約や元テキストを取り出し、LLMに渡す
4. LLMが以下を統合して、回答を生成
   - エンティティの説明
   - そのエンティティが行った行動
   - 関係する組織・イベント
   - 出典となるニュース記事断片

Microsoftのブログ例だと、「NovorossiyaがATM爆破などの破壊活動と関連づけられている」という結論を、グラフ上のエッジ（Novorossiya → 事件 → PrivatBankなど）と原文スニペットを根拠として説明していました。

#### (3) DRIFT Search（グローバルとローカルのハイブリッド）

多くの実用的な質問は、「全体像と詳細の両方」が必要です。
そこで導入されたのが DRIFT（Dynamic Reasoning and Inference with Flexible Traversal）です。

**典型的な流れ：**

1. **Primerフェーズ（グローバル）**
   - 質問をコミュニティレポートに当てて、大まかな回答と、LLMが考えるフォローアップ質問を生成
   - 例：「FDAとCDCが2023年11月のリンゴシナモン製品の鉛汚染にどう対応したか？」
2. **Follow-upフェーズ（ローカル）**
   - フォローアップ質問を、ローカル検索でそれぞれ解く
   - FDA周辺のローカルグラフ → 具体的な発表・リコール情報
   - CDC周辺のローカルグラフ → 健康勧告・ガイドライン
3. それらの結果を統合して、最終回答を出す

こうすることで、「広くテーマを捉えつつ、細部も正確に」という動的な探索が可能になります。

### GraphRAGと従来RAGの本質的な違い（メカニズム観点）

メカニズムを整理すると、大きく以下が違います。

#### 1. 検索空間の単位

- **従来RAG：**
  - 単位は「テキストチャンク」
  - クエリとチャンクのベクトル類似度でランキング
- **GraphRAG：**
  - 単位は「エンティティ」「関係」「コミュニティ」「要約レポート」
  - クエリをグラフ構造にマッピングし、関連コミュニティとローカルサブグラフを選んでから、その周辺のテキストを読む

→ ベクトルRAGは「点と点の距離」で考え、GraphRAGは「グラフの構造と階層」を見てから読むイメージです。

#### 2. 文脈の集め方

- **従来RAG：**
  - クエリに似たチャンクを数個取り、LLMに一気に渡す
  - その中に必要な情報が無いと、「その情報はありません」となる
- **GraphRAG：**
  - クエリに関連するエンティティ・コミュニティから、関連する周辺テキストや要約を段階的に集める
  - その結果、「Novorossiyaそのものを直接書いているチャンク」が無くても、間接的な関係（テロ計画の記事など）を辿って答えを導ける

#### 3. グローバルな構造把握

- **従来RAG：**
  - コーパス全体を俯瞰する構造は無く、「局所的な類似チャンクの集まり」のみ
- **GraphRAG：**
  - コミュニティ検出＋階層構造＋コミュニティ要約により、「このデータセットにはこういうテーマの塊がある」という全体構造を明示的に持つ

→ これが「query-focused summarizationを大規模に扱える」という論文の主張につながっています（arXiv:2404.16130）。

#### 4. プロベナンスと評価

GraphRAGは、各主張が以下のどれに支えられているかを追跡しやすい設計になっています。

- どのエンティティ
- どの関係
- どのコミュニティ
- どの原文テキスト

そのため、

- SelfCheckGPTのようなfaithfulness評価
- 「どのニュースソースから来た情報か？」の追跡
- 論点ごとの出典比較

などがやりやすく、企業利用の監査性や説明可能性の観点で大きなメリットがあります。

### LazyGraphRAGとの違い（実装・コスト観点の補足）

実装側を触るなら、フルGraphRAGとLazyGraphRAGの違いも重要です。

- **フルGraphRAG：**
  - インデックス時にLLMでエンティティ要約・コミュニティ要約を全部作っておく
  - クエリ時はそれを再利用するので高速・高品質
  - 代わりに、インデックスコストがベクトルRAGの100〜1000倍レベルに膨らむ
- **LazyGraphRAG：**
  - インデックス時はLLMをほぼ使わず、軽量な句・名詞句抽出などでグラフだけ作る
  - コミュニティ要約などはクエリ時に必要な分だけLLMで生成
  - インデックスコストはフルGraphRAGの0.1%程度、クエリ時の探索は少し複雑になるが総合的に安い＆高品質

どちらも「知識グラフ＋コミュニティ構造＋グラフベース検索」という本質は同じで、「いつLLMに要約させるか（オフラインかオンラインか）」が違う、という理解で良いと思います。

### まとめ（メンタルモデル）

- **GraphRAGは、** 以下のパイプラインで動く「グラフベースRAG」
  - 文書群 → LLMでエンティティ・関係抽出 → 知識グラフ → コミュニティ検出 → コミュニティ要約
  - クエリ → グラフ上のエンティティ／コミュニティにマップ → 関連サブグラフ＋要約から回答
- **メカニズムのキモは、** 以下の3点
  - 構造化（グラフ化）してから読む
  - 階層的コミュニティ要約を作って、局所〜全体まで多層の視点を持つ
  - クエリに応じて適切なコミュニティ・サブグラフを動的に選択する
- **その結果、** 以下が必要な問いに強くなる
  - マルチホップ推論
  - データセット全体のテーマ抽出
  - 「点」ではなく「ネットワーク」としての理解

### 全体像：GraphRAG の「重さ」をどう減らすか

まず大きな構図から整理すると、最近の GraphRAG 系の研究・実装はだいたい次の三つのコスト源をどう潰すか、という話になっています。

1. **インデックス構築時の LLM コスト** ── 各チャンクからエンティティ・関係・コミュニティ要約を LLM で生成する部分。
2. **グラフ構築そのものの計算コスト／メモリ** ── 知識グラフ構築、Leiden などのコミュニティ検出、巨大なエッジリストの保持。
3. **クエリ時（検索＋生成）のトークン・レイテンシ** ── どれだけ多くのサマリ・チャンクを LLM に食わせるか、どれだけ段階的に呼ぶか。

ここで取り上げる論文・記事は、この三つをそれぞれ違うやり方で削っています。

- **GraphRAG-V** …「グラフを捨ててベクトルコミュニティに寄せる」ことで構築・検索を高速化。
- **LazyGraphRAG** …「インデックスでは LLM を一切使わず、クエリ時に必要な分だけ LLM を遅延使用」してコストを制御。
- **kk-core ベース GraphRAG** …「Leiden を捨てて kk-core 由来の階層クラスタ＋サンプリングで、同じ GraphRAG をもっと安く安定に」。

順番に、構造とコスト削減メカニズムを整理していきます。

### GraphRAG-V：トリプルなし・グラフなしの「ベクトルコミュニティ GraphRAG」

#### 発想の転換

元の Microsoft GraphRAG は、

- LLM でエンティティ・リレーションを抽出 → KG を構築
- Leiden でコミュニティ検出
- 各コミュニティを LLM で要約（多段階）

というかなりヘビーなパイプラインでした。

GraphRAG-V が主張しているのは、ざっくり言うと**「性能向上の本質は triple-KG ではなく、"チャンクのコミュニティ構造" にある。なら、トリプルも KG も作らず、チャンク embedding だけでコミュニティを作ればいい」**という考え方です。

#### アーキテクチャ（Indexing 側）

論文の流れを要約すると、Indexing は次のようになります。

1. コーパス → 600 token + 100 token overlap でチャンク化
2. bge-m3 で全チャンクを一度だけ埋め込み → 単位ベクトルに正規化
3. FAISS HNSW に格納（ベクトル RAG 用）
4. 同じ埋め込み行列に対して VLouvain を実行
   - 内積ベースの類似度を用い、「暗黙の完全グラフ」を仮定したまま Louvain 相当のモジュラリティ最大化を行う
   - ここがポイントで、エッジリストを明示的に構築しない（graph-free community detection）
5. 各レベルのコミュニティごとに、非自明なものだけ LLM で 2文サマリを生成

結果として、永続的に持つのは

- チャンク埋め込み＋FAISS インデックス
- 各レベルのコミュニティ割り当て（`partitions[L]`）
- コミュニティサマリ（短い JSON）

だけで、トリプルやエッジリストは一切保存しません。

これにより：

- **グラフ構築の計算量・メモリ：** O(n²) のエッジ構築 → O(nd) の行列演算に圧縮
- **インデックス時 LLM コール：** triple 抽出ゼロ、コミュニティサマリ 1 パスだけ

実験では MultiHopRAG コーパスに対して、

- ベクトルインデックス 74 秒
- VLouvain 5.1 秒
- コミュニティサマライズ 4 分
- → **GraphRAG-V 全体で約 5.3 分**（対して Microsoft GraphRAG は約 3 時間）

#### アーキテクチャ（Query 時）

Retrieval も元 GraphRAG をなぞりつつ、すべてベクトル空間で完結するように設計されています。

- **Local 検索**
  - クエリを 1 回だけ埋め込み
  - 最下層（fine-level）のコミュニティサマリと cosine 類似度を取り、上位 kc コミュニティを選ぶ
  - そこに所属するチャンクの埋め込みをクエリと再度比較し、上位 np パッセージだけを選んで LLM へ
- **Global 検索**
  - 同じクエリベクトルを、今度は最上位の root コミュニティサマリと比較し、上位 3 root を選ぶ
  - 各 root の直下の child サマリをまとめて LLM に渡し「部分回答」を生成
  - 複数の部分回答を再度 LLM に渡して、最終回答を fuse

Embedding はクエリごとに 1 回だけですし、グラフ探索はなく、すべて「ベクトル間類似度＋事前計算された階層構造」で済ませています。

#### コストと品質

MultiHopRAG 全 2,556 問に対する実験では、

- VectorStore-RAG（Qwen2.5）: Recall 37.9% → Accuracy 44.6%
- GraphRAG-V（Local+Global）: Recall 48.8% → Accuracy 49.4%

と、Recall +11pt・Accuracy +約4pt 改善しつつ、

- 全問回答時間: Vector RAG 12 分 → GraphRAG-V 42 分（GraphRAG Local／Global の推定は 28h / 144h）

なので、

- Indexing コストは GraphRAG より 30〜40 倍速い
- クエリレベルでは VectorRAG より重いが、GraphRAG よりは二桁以上軽い
- 同じ GraphRAG 系として見ると、「トリプルを捨てたベクトルコミュニティ版」が最も現実的、というポジション

### LazyGraphRAG：インデックスは VectorRAG 並、グラフは「クエリ時にだけ深く見る」

#### 背景：GraphRAG の重さと更新問題

Microsoft / Qiita の解説が強調している GraphRAG の課題はだいたい共通で、

- **インデックス時に：**
  - 全チャンクに対して LLM でエンティティ・関係抽出（JSON スキーマ）
  - コミュニティごとに LLM 要約
  - → 大量の LLM 呼び出しと時間・コスト
- **ドキュメント追加時に：**
  - グラフ全体の再構築・再サマライズが必要で、インクリメンタル更新がほぼ不可能

という点です。

LazyGraphRAG は、ここに対して

- インデックス構築では LLM を一切使わない（VectorRAG 並みに安い）
- その代わり、クエリ時に必要なコミュニティだけを「遅延的に」LLM で評価・要約する

というアーキテクチャを取ります。

#### Indexing：LLM ゼロで「コンセプトグラフ」だけ作る

MS の公式記事・Qiita 記事からまとめると、LazyGraphRAG の Indexing はこうです。

1. ドキュメントをチャンク化
2. NLP（SpaCy の名詞句抽出など）で名詞句（concept）を抽出
3. 名詞句の共起から**概念共起グラフ**を作る（PMI など）
4. この概念グラフに対して Leiden などでコミュニティ検出
5. ここまで一切 LLM を使わない（= VectorRAG と同等のインデックスコスト）

GraphRAG のような「エンティティ・リレーション triple グラフ」ではなく、**concept 共起グラフ**である点がポイントです。

#### Query 時：Best-first × Breadth-first × Iterative Deepening

クエリ時のパイプラインが LazyGraphRAG の肝です。公式記事と Qiita 実装から整理すると、

1. **QueryExpander**
   - クエリを LLM で分解・拡張し、複数のサブクエリにする
2. **ベクトル検索＋コミュニティランキング（best-first）**
   - サブクエリごとに、通常のベクトル検索でチャンクを取ってくる
   - 「どのコミュニティに属するチャンクが上位に来ているか」で、コミュニティをランク付け
3. **RelevanceTester（breadth-first）**
   - コミュニティ順に、まだ評価していないチャンクを少しずつ取り出し、LLM で関連度を 0–10 などでスコアする
   - これを「relevance_test_budget（予算）」の範囲内で繰り返す
4. **IterativeDeepener（反復深化）**
   - 関連度の高いチャンクについて、ClaimExtractor で「主張・クレーム」を抽出
   - クレーム数や予算を見ながら、コミュニティ探索を少しずつ深くする
5. **LazyContextBuilder**
   - 集めたクレームと関連文から、優先度の高いものだけを選び、LLM のコンテキスト上限内に収まるような「最終コンテキスト」を組む
6. 最後に LLM で回答を生成

といった「クエリごとに小さなグラフ探索＋ LLM 評価を行う」設計です。

重要なのは、`relevance_test_budget`（公式では 100 / 500 / 1500 など）を主なノブとして、

- 予算が小さい → LLM 評価回数が少なく高速・低コスト
- 予算を増やす → コミュニティやチャンクを広く深く見にいく

という**コスト vs 品質のトレードオフを一つのパラメータで制御できる**ようにしている点です。

#### コスト削減の実績

MS 公式ベンチマーク（AP News 5,590 記事・100 クエリ）では、ざっくり次のような数字が出ています。

- **インデックスコスト**
  - GraphRAG: 基準
  - LazyGraphRAG: GraphRAG の 0.1%（= VectorRAG 並）
- **クエリコスト**
  - GraphRAG Global C2: 基準
  - LazyGraphRAG Z500: GraphRAG Global の約 4% のコストで、局所／全体クエリとも品質で上回る

Qiita の実装＋ベンチマーク（100万チャンク・1000クエリ）でも、

- 各クエリ約 3 秒
- データ量 200 倍→処理時間 182 倍（ほぼ線形スケール）
- 従来 GraphRAG と比べて「約 1/100 コストで同等品質」

という結果が報告されています。

#### LazyGraphRAG の位置付け

まとめると LazyGraphRAG は：

- Indexing は VectorRAG と同じくらい軽い（LLM なし）
- Graph 構造は「コンセプトグラフ＋コミュニティ階層」として事前に用意
- クエリ時にだけ、そのグラフを「最小限の LLM 予算ですこしずつ探索」する

という形で、**GraphRAG のグローバル文脈の強さ × VectorRAG のコスト感**を、`relevance_test_budget` という一つのパラメータで調整可能にしたアーキテクチャだと捉えると分かりやすいと思います。

### kk-core 分解を用いた GraphRAG：Leiden の不安定さとコストをどう抑えるか

#### 問題設定：Leiden + モジュラリティは sparse KG に向かない

arXiv の「Core-based Hierarchies for Efficient GraphRAG」は、既存の GraphRAG（Edge et al.）が前提にしている Leiden ベースのコミュニティ検出が、知識グラフのような疎なグラフには根本的に向いていない、というところから出発しています。

ポイントは三つです。

1. 知識グラフは平均次数が 3〜4 程度の「疎グラフ」で、次数 1 のノードが 50〜60% を占める。
2. そのようなグラフでは、モジュラリティ最大化の解が指数的に多数存在し、最適解周りが「高原（degeneracy）」になる（Good et al. 2010 の結果）。
3. 結果として Leiden は、初期値やランダムシード、エッジのわずかな変化でコミュニティが大きく変わり、「再現性が低く・粒度もバラバラ」な階層ができてしまう。

論文ではこれを定理として形式化し、「疎な KG に Leiden を当てると、構造的に非再現的なコミュニティを延々と再計算することになる」と主張しています。

#### 解決策：Leiden → kk-core（k-core）の階層に置き換える

そこで提案しているのが、モジュラリティ最適化を捨てて、kk-core 分解に基づく階層を使う GraphRAG です。

- **kk-core 分解とは**
  - 各ノードについて「そのノードが属しうる最大の k-core（最小次数 k の誘導部分グラフ）」を求める手続き
  - 計算量は O(|E|)（一次スキャンで済む）
  - 1-core ⊇ 2-core ⊇ 3-core … という決定的・ネストした階層が得られる
- **知識グラフとの相性**
  - 高い core number のノードは「多様なパスで他のノードと結ばれている」＝トピックの中心的なエンティティになりやすい
  - Triangles ベースの k-truss は、KG が triangle-poor なので不適（3-truss ですぐにスカスカになる）→ degree ベースの kk-core がちょうどいい

これを GraphRAG に組み込むと：

- コミュニティ境界が**決定的（deterministic）**で、再現性が高い
- モジュラリティ最大化のような大域最適化をしないので計算が圧倒的に軽い
- 「2-core は骨格、1-core 周縁は residual」といった扱いが自然にできる

というメリットがあります。

#### 実際のヒューリスティクス：RkH / M2hC / MRC + RRTC

論文では kk-core 単体ではなく、実用上の問題（極端に小さいクラスタ・トークン制約など）に対応するためにいくつかのヒューリスティクスを重ねています。

- **RkH（Residual-aware kk-core Hierarchy）**
  - 各レベル ℓ の時点で、core 部分 Sc と residual 部分 Sr に分ける
  - 大きすぎるコンポーネントは Split で最大サイズ M を超えないよう分割
  - サイズ 1 のクラスタ（singletons）を 2-hop 接続に基づいて束ねる
  - 最終的に、**「サイズ制限付き・連結性を保ったクラスタ集合」**を得る
- **M2hC / MRC（小クラスタのマージ）**
  - 2-hop クラスタや residual クラスタのうち、サイズ 2 のものを近いクラスタにマージ
  - あまりにも小さなクラスタが多いと、GraphRAG のコミュニティ単位の評価で無視されがちなので、それを防ぐ
- **RRTC（Round-Robin Token-Constrained Selection）**
  - Leaf レベルのクラスタに対し、エッジを「ノード次数に基づいて」ランク付け
  - 各クラスタから少しずつエッジをサンプリングしながら、トークン予算に到達するまで round-robin で選択
  - これにより、トークン消費を 40% 近く削りつつ、GraphRAG C2/C3 と同程度の comprehensiveness/diversity を維持できると報告

#### 結果とコスト面

GPT-3.5, GPT-4o-mini, GPT-5-mini を evaluator にした head-to-head 比較では、

- Comprehensiveness / Diversity で C2/C3（Leiden GraphRAG）を体系的に上回る
- 特に M2hC-LF（leafレベル＋2-hop マージ）が一貫して強い
- 同じデータで、コミュニティ数・トークン利用量が目に見えて減る
  - 例：news データセットで C3: coverage 約 71% / RkH-LF: 約 76% / MRC-LF: 約 55%（かなり絞り込み）

つまりこのラインは、**GraphRAG のパイプライン自体は維持しつつ、「コミュニティ階層の構築」と「トークン消費」を kk-core＋ヒューリスティックで安く・安定にする**というアプローチです。GraphRAG-V や LazyGraphRAG のように「トリプル・KG 自体を捨てる」ほどの割り切りはせず、元の GraphRAG の枠内で最適化している点が対照的です。

### 他の軽量化・拡張アプローチ（ざっくり）

ここまでの記事群には、他にもいくつか関連する方向性が出てきます。

- **LightRAG**
  - GraphRAG の「インクリメンタル更新ができない」「1 クエリで数十万トークン消費する」という弱点を意識
  - ベクトル検索＋軽量なグラフトラバースの二段構成＋真のインクリメンタル更新を実現したフレームワーク
  - GraphRAG と同等の精度で、1 クエリあたり 100 トークン未満の retrieval に抑える、という結果が報告されている
- **PageIndex / PageIndex ベースの vectorless RAG（Tao An の Medium 記事）**
  - 「embedding や vector DB に頼らず、文書の構造（目次ツリー）に沿って LLM をナビゲートさせる」系
  - FinanceBench で 98.7% の高精度だが、ツリー探索で多数の逐次 LLM コールを行う＝レイテンシ・コストが重いというトレードオフがあり、Tao の記事では「効率指標を出さないのは問題」と批判的
- **CausalRAG / Hypergraph RAG など**
  - グラフをさらに「因果グラフ」や「ハイパーグラフ」に拡張し、多段推論・ global sensemaking を強化する方向
  - ただし現時点では、GraphRAG よりさらに重い・実装複雑、という傾向が強く、「コスト削減・軽量化」という観点ではむしろ逆方向の研究と言える

---

### どういう文脈でどれを選ぶか（ざっくり指針）

最後に、実務的な観点で三つの主役を並べてみると：

- **GraphRAG-V**
  - すでに VectorRAG を持っていて、「もう少し multi-hop を取りたい」ケース
  - トリプル KG や重いグラフ実装に踏み込まず、ベクトルコミュニティの追加で済ませたいときの現実解
- **LazyGraphRAG**
  - インデックスは頻繁に更新される（論文が毎日増えるなど）
  - コスト制約がシビアで、**「インデックス構築に LLM を使う余裕がない」**組織
  - 1 クエリあたりのコストを「予算パラメータ」で明示制御したいとき
- **kk-core ベース GraphRAG**
  - すでに GraphRAG（Edge et al. 系）のパイプラインを持っている
  - Leiden の不安定さやコスト、トークン消費をどうにかしたい
  - ただし「GraphRAG 自体は捨てたくない」場合の**内部最適化路線**

Tao An の記事が強調していた通り、「一番精度が出るアーキテクチャ」＝「現場で一番よいアーキテクチャ」ではなく、レイテンシ・トークンコスト・運用複雑性とのトレードオフが本質になっています。

今回の三つは、それぞれ違うレイヤで GraphRAG の重さを削るアプローチなので、自分のワークロード（更新頻度、許容レイテンシ、月間トークン予算）を想像しながら**「どこを捨てるか」**を設計するイメージで見ると整理しやすいと思います。

## 10: 2026/3/21 10:00

> MemGPT論文の要旨

この論文は、LLMを「OS風のメモリ管理システム」で拡張し、有限のコンテキストでも”無限コンテキストっぽく”使えるようにする仕組み（MemGPT）を提案しています。

### 全体像：何をした論文か

この論文のテーマは「コンテキストウィンドウが有限なLLMを、どうやって長期対話や超長文ドキュメント解析に耐えさせるか」です。
単にコンテキスト長を伸ばすのではなく、OS の仮想メモリのように「ページング」しながら LLM 自身にメモリ管理をさせるという発想で設計されたシステムが MemGPT です。

アイデアとしては：

- LLMのコンテキスト = RAM（主記憶）
- 外部ストレージ（DBなど） = ディスク
- LLMに「関数呼び出し機能」を使わせて、必要な情報を外部から読み書きさせる
- その結果、見かけ上「無限に近いコンテキスト」を持つエージェントとして振る舞える

という構図です。

### MemGPT のアーキテクチャの要点

#### 1. メモリ階層：Main context と External context

MemGPTは、LLMの扱う情報を二つに分けています。

**Main context（主記憶）**

LLMのプロンプトとして実際に入るトークン領域。ここにあるものだけをLLMは「今見えているコンテキスト」として扱う。

- **System instructions（システムプロンプト）**
  MemGPTの使い方、メモリ階層の説明、関数の仕様など。読み取り専用。
- **Working context**
  固定サイズの読み書き可能ブロック。
  ユーザのプロフィール・好み・重要な出来事など「長期的に持っていたい要約された情報」を置く想定。
- **FIFO Queue（キュー）**
  直近のメッセージ履歴（ユーザ発話・エージェント発話・システム警告・関数の入出力など）を時系列で格納するリングバッファ。
  古いメッセージは追い出され、その代わりに「追い出された内容の再帰的サマリ」が最初のシステムメッセージとして残ります。

**External context（外部コンテキスト）**

LLMのコンテキスト外にあるデータ群。

- **Recall storage**
  メッセージ履歴を保存しておくDB。キューから溢れた会話もここに残り、検索で再取得可能。
- **Archival storage**
  文書やWikipedia記事のような長文データを格納しておくDB。ベクトル検索（pgvector + HNSW）で類似文書を取ってくる。

ポイントは、External context にある情報は、関数呼び出しで Main context に「明示的に持ってこない限り」LLM には見えないという設計です。

#### 2. Queue Manager：コンテキスト圧迫の監視と退避

Queue Manager は、Main context のうち FIFO キュー部分を管理するコンポーネントです。

- 新しいユーザメッセージ・LLM出力が来るたびにキューに追加
- その内容を同時に **Recall storage（メッセージDB）** にも書き込む
- コンテキストの長さがしきい値に近づいたら：
  - “memory pressure” のシステムメッセージを挿入
  - LLMに「そろそろ重要情報を working context や外部ストレージに退避した方がいい」と気付かせる
- コンテキストが限界を超えたら：
  - FIFO キューの古いメッセージをまとめて退避
  - それらを元に「再帰的サマリ」を更新し、古い詳細はRecall storageへ保存

図1・図2の例では、
ユーザとの雑談の中で「誕生日が2月7日」「彼氏の名前がJames」などを working_context に書き出したり、
後で「Six Flags」という話が出てきた時に recall_storage を検索して過去の会話を引っ張ってきたりしています。

#### 3. Function Executor：LLM出力を「関数呼び出し」として解釈

MemGPTは、LLMの出力トークンを「自然言語として読む」のではなく、「関数呼び出しとしてパース」します。

- LLMは「関数スキーマ」を事前に教えられている（例：`recall_storage.search(...)`, `working_context.append(...)` など）
- 出力をパースし、正しい形式なら該当関数を実行
- 結果やエラーは再びMain contextにメッセージとして追加され、次のLLM推論に使われる

ここで重要なのが **function chaining（関数の連鎖呼び出し）** です。

- 関数呼び出し時に `request_heartbeat=true` というフラグを付けると：
  - 関数実行後、すぐに次のLLM推論を続行（ユーザ入力を待たずに、連続的に複数の関数を呼び出せる）
- これにより、以下のような「マルチステップ処理」が可能になる：
  - 検索結果のページング
  - 複数ドキュメントの段階的な読み込み
  - ネストしたKVの多段階ルックアップ

#### 4. Control Flow：イベント駆動で動く”LLM OS”

MemGPTは「イベント駆動」でLLMを呼びます。

- イベントの例：
  - ユーザメッセージ
  - コンテキストが溢れそうだというシステム警告
  - 「ユーザがログインした」「ドキュメントのアップロードが完了した」などの外部イベント
  - 定期実行イベント（スケジューラ）で、ユーザ操作なしでも自走させられる

イベントをテキストメッセージに変換し、Main context に追加した上で推論を走らせます。
LLMが関数を呼ぶか、ユーザへ応答を返すか、チェーンを続けるか…といった制御も、この枠組みの中で行われます。

### 実験：MemGPTは何がどのくらい良くなるか

論文では、(1) 長期対話エージェント と (2) ドキュメント解析 の2ドメインで評価しています。

#### 1. 長期対話（Multi-Session Chat 拡張）

基礎となるデータセットは Multi-Session Chat (MSC) で、同じ二人が5セッションに渡って会話し続けるログです。
これを拡張して、以下の2タスクを作っています。

**(a) Deep Memory Retrieval (DMR)：一貫性の評価**

- 6回目のセッションで、「過去の会話を覚えているか？」を問う質問（Q&A）を追加
- 質問と正解（狭い答えの範囲）を別のLLMで生成
- 評価：
  - ROUGE-L
  - GPT-4によるジャッジ（正解と同じトピックを含んでいるか）

結果（Table 2）：

- ベースモデル単体（GPT-3.5 / GPT-4 / GPT-4 Turbo）は、過去会話の要約しか見られないため精度が低い
- MemGPT + GPT-4 / GPT-4 Turbo は Accuracy 90%超と大幅に向上
- 理由：過去の会話全体がRecall storageにあり、必要に応じて検索・ページングしてMain contextに再挿入できるため

**(b) Conversation Opener Task：エンゲージメントの評価**

- 新しいセッションの最初の「一言目（オープナー）」をどれだけ”その人らしく・魅力的に”書けるかを評価
- 評価は CSIM（コサイン類似度）で：
  - ペルソナ情報との類似度（SIM-1, SIM-3）
  - 人間が書いたオープナーとの類似度（SIM-H）

結果（Table 3）：

- MemGPT は、人間が書いたオープナーと同等〜それ以上のスコアを出すこともある
- Working context に蓄えたユーザ情報（好み・イベントなど）を活用して、よりパーソナライズされた文面を生成できるため

#### 2. ドキュメント解析

**(a) Multi-Document Question Answering**

- NaturalQuestions-Open のQA問題とWikipediaドキュメントを使ったRetrieval-Reader型タスク
- Retrieverはembedding（text-embedding-ada-002）で上位K文書を取得
- 比較：
  - 固定コンテキストLLM（GPT-3.5 / 4 / 4 Turbo）：retrieverで取ってきたK文書を直接プロンプトに詰め込む
  - MemGPT：
    - Wikipedia全体を Archival storage に投入
    - LLMが自分で `archival_storage.search` しながら必要な文書をページングして読む

結果（Figure 5）：

- 固定コンテキストLLMは、Kが増え、コンテキストが埋まるにつれて限界が来る
  - さらにKを増やすと、文書を切り詰めざるを得ず、精度が落ちていく
- MemGPTは、コンテキスト長に依存せず、必要なだけretrievalを繰り返せるので、性能が頭打ちになりにくい
- ただし embedding 検索の限界（ゴールド文書が上位に出てこない）に引きずられるため、この部分はRAG全般の課題

**(b) Nested Key-Value Retrieval：マルチホップ推論の検証**

- syntheticなUUIDのKey-Value集合を用いたタスク
- 今回の拡張版では「値もキーになり得る」ため、例：`key1 -> key2 -> key3 -> value` のような多段階ルックアップ（マルチホップ）が必要

評価：

- 総ペア数は約8kトークン分（GPT-4のコンテキスト程度）
- ネストレベルを0〜4まで変え、30パターンずつ検証

結果（Figure 7）：

- GPT-3.5 / GPT-4 / GPT-4 Turbo 単体は、ネストが1〜2段階を超えるとほぼ全滅
  - 原因：きちんと段階的にルックアップを繰り返さず、途中で元の値を返してしまうなど
- MemGPT + GPT-4 は、ネストレベルが増えても精度が落ちず、4段階でもほぼ完答
  - 関数呼び出しで `archival_storage.search` を何度も呼び、「値がまだキーなのか」を確認し続けるというループを設計どおりに実行できている
- GPT-4 Turbo / GPT-3.5 を基盤にした MemGPT も改善するが、関数呼び出しの扱いの弱さで途中から精度低下

### 位置づけと関連研究との違い

論文後半では、MemGPT を以下のラインと比較して位置づけています。

- **Long-context LLM**
  - Longformer, Reformer, Linformer, positional interpolationなど
  - これらは「コンテキスト長そのものを伸ばす」アプローチだが、計算コストや、長文中盤のトークンへの注意が薄くなる問題もある
- **Retrieval-augmented models (RAG)**
  - DPR, RAG, FLARE など
  - 「外部知識ベースから必要な情報を取ってくる」のは共通だが、MemGPTはより体系的に「OS風」に整理している：
    - メモリ階層（Main/Recall/Archival）
    - イベント駆動の制御
    - 自律的なメモリ編集（working context更新やpersonal memoryの保持）
- **LLMエージェント**
  - ReAct、WebGPT、Generative Agents など
  - 行動・計画・ツール使用に焦点を当てた研究に対し、MemGPTは **”長期メモリとコンテキスト管理”にフォーカス** している

### まとめ：MemGPTの本質的な貢献

論文の結論部で強調されているポイントを整理すると：

- LLMのコンテキストウィンドウを、そのまま「OSにおける物理メモリ」とみなして扱う
- 「どの情報を in-context に置き、どれを外に追い出し、いつまた読み込むか」という問題を、以下で実現した：
  - メモリ階層（working context / FIFO queue / recall / archival）
  - 関数呼び出し
  - イベント駆動・関数チェーン
- その結果、既存の固定コンテキストLLMの上に以下を構築できることを実験で示した：
  - 長期対話で「覚えている」エージェント
  - 大規模ドキュメントを超えて推論できるエージェント

将来の方向性としては：

- 他ドメイン（巨大ログ解析、長期プロジェクトマネジメントなど）への展開
- メモリ階層の実装により高度なDB・キャッシュ・ストレージを組み合わせること
- メモリ管理ポリシー（何をどのタイミングで保持・破棄するか）の改良

などが挙げられています。

### 補足：エンジニア視点で見ると

システムエンジニア的に見ると、MemGPTは以下の構成になっています：

- LLMを「CPU」兼「MMU（メモリ管理ユニット）」として動かす
- 外部のストレージ（PostgreSQL + pgvector）を「仮想メモリ」として扱う
- 「関数呼び出し = システムコール」としてOSライクなAPIを与える

かなりOS/ミドルウェア寄りの発想です。

LLMそのものを改造せず、「プロンプト＋関数呼び出し＋外部ストレージ＋制御ロジック」で長期メモリをエミュレートしているので、
既存のAPIベースLLMに対しても比較的実装しやすいアーキテクチャという点も実務的には大きなポイントだと思います。
